# Day 6 PRD：小程序 AI 点单页面 MVP

## 1. 文档信息

- 文档名称：小程序 AI 点单页面 MVP PRD
- 所属项目：AI 咖啡点单与知识助手
- 对应计划：Day 6｜小程序 AI 点单页面
- 文档版本：v1.0
- 当前阶段：开发前需求确认

## 2. 背景与目标

### 2.1 背景

项目已在 Day 3-5 完成 AI 点单后端能力：

- 已新增 `POST /api/ai/order-assistant/` 接口。
- 后端可接收用户自然语言点单需求。
- 后端已接入 DeepSeek，并要求 AI 返回结构化 JSON。
- 后端已校验推荐结果中的 `product_id` 是否匹配真实商品。
- 后端已支持 AI 异常、非法 JSON、商品不匹配等 fallback 兜底。

Day 6 需要将接口能力产品化，在微信原生小程序中新增正式 AI 点单页面，让用户可以输入一句自然语言需求，并看到清晰的推荐商品卡片。

### 2.2 产品目标

完成一个可演示的 AI 点单页面 MVP，实现：

```text
用户从点单页进入 AI 点单页
→ 输入饮品需求
→ 点击发送
→ 页面展示 loading
→ 调用 AI 点单接口
→ 展示 1-3 个推荐商品卡片
```

### 2.3 非目标

本阶段不实现以下内容：

- 不支持推荐商品加入购物车，留到 Day 8。
- 不支持一键直接加入购物车；后续应进入原有客制化选择流程后再确认加入。
- 不改造后端 AI 点单接口字段结构。
- 不做多轮对话记忆。
- 不做快捷需求按钮，留到 Day 9。
- 不新增底部 TabBar 入口。
- 不做复杂推荐解释链路，只展示简洁推荐理由。

## 3. 用户与场景

### 3.1 目标用户

- 普通点单用户：不知道喝什么，希望快速获得推荐。
- 咖啡社团成员：希望根据温度、甜度、奶源、提神等偏好获得更合适的饮品建议。
- 项目演示观众：老师、同学、面试官，用于理解 AI 如何接入真实点单流程。

### 3.2 核心场景

用户进入点单页时看到 AI 点单入口，点击进入 AI 点单页。用户输入“我想喝冰的、低糖、不要牛奶的咖啡”，系统调用 AI 点单接口，并展示推荐商品，如冰美式、冷萃等，同时说明推荐理由和建议规格。

## 4. 入口设计

### 4.1 入口位置

入口放在现有小程序点单页 `pages/menu/menu` 中。

### 4.2 入口交互

- 在点单页增加一个 AI 点单入口区域或按钮。
- 用户点击入口后，使用 `wx.navigateTo` 跳转到 `pages/ai-order/ai-order`。
- 未登录用户也允许进入 AI 点单页，用于先了解功能价值。

### 4.3 入口文案建议

- 主标题：AI 帮我点
- 副文案：说出口味、温度或预算，让 AI 推荐适合你的咖啡
- 按钮文案：去试试

## 5. 页面功能需求

### 5.1 页面路径

新增微信小程序页面：

```text
pages/ai-order/ai-order
```

需要新增文件：

```text
frontend/pages/ai-order/ai-order.wxml
frontend/pages/ai-order/ai-order.wxss
frontend/pages/ai-order/ai-order.js
frontend/pages/ai-order/ai-order.json
```

同时需要在 `frontend/app.json` 的 `pages` 中注册该页面。

### 5.2 页面结构

页面包含以下模块：

1. 顶部说明区
   - 展示 AI 点单助手标题。
   - 展示简短说明，降低用户输入成本。

2. 输入区
   - 一个多行输入框。
   - 支持用户输入自然语言需求。
   - 输入为空时不能发送，并给出提示。

3. 发送按钮
   - 文案：开始推荐 / 让 AI 推荐。
   - 请求中禁用按钮，避免重复提交。

4. loading 状态
   - 点击发送后展示“正在分析你的口味...”。
   - 请求完成后关闭 loading。

5. 推荐结果区
   - 展示 1-3 个推荐商品卡片。
   - 每张卡片展示商品图片、商品名、价格、推荐理由、规格建议。
   - 可预留“选择规格”按钮样式，用于表达 Day 8 的后续闭环，但 Day 6 暂不绑定真实加购逻辑。

6. 兜底提示区
   - 当后端返回 fallback 推荐时，展示友好提示。
   - 当 AI 无法推荐或用户输入无关内容时，展示可行动的引导文案。

### 5.3 输入框需求

- 字段名称：`userInput`
- 支持多行输入。
- placeholder 示例：

```text
比如：我想喝冰的、低糖、不要牛奶的咖啡
```

- 输入建议：
  - 温度：热的、冰的、少冰
  - 甜度：低糖、无糖、正常甜
  - 风味：奶香、清爽、苦一点、酸一点
  - 需求：提神、下午喝、预算 20 内

### 5.4 发送规则

点击发送时按以下规则处理：

1. 如果用户未登录：
   - 弹窗提示“请先登录后再使用 AI 点单助手”。
   - 用户确认后跳转到登录页。

2. 如果输入为空：
   - 使用 toast 提示“请先告诉我你想喝什么”。
   - 不发起接口请求。

3. 如果已登录且输入不为空：
   - 设置 loading 状态。
   - 调用 `POST /api/ai/order-assistant/`。
   - 请求参数包含 `user_input` 和 `user_id`。

### 5.5 商品信息补齐规则

Day 6 不修改后端 AI 接口。AI 接口返回的推荐项只作为推荐逻辑结果使用，商品价格和图片由前端通过已有商品列表补齐。

页面进入时或发送前，需要调用：

```text
GET /api/products/
```

前端根据 AI 返回的 `product_id` 与商品列表中的 `id` 匹配，补齐：

- `price`
- `image`
- `name`
- `description`

如果商品列表匹配失败：

- 仍可展示 AI 返回的 `product_name` 和 `reason`。
- 价格展示为“价格以菜单为准”或隐藏价格。
- 图片展示默认占位图。

### 5.6 客制化选项与推荐逻辑

AI 点单推荐需要同时关注“商品选择”和“规格建议”，但二者不能混为一谈。

#### 5.6.1 商品推荐原则

AI 应先判断用户需求是否影响商品本身：

- “低糖 / 无糖”通常优先映射为甜度规格，不一定改变商品类型。
- “不要牛奶 / 无奶 / 清爽”会影响商品筛选，应优先推荐美式、冷萃、手冲等非奶咖商品。
- “奶香 / 顺滑 / 像拿铁”会影响商品筛选，应优先推荐拿铁、澳白、风味奶咖等商品。
- “提神 / 咖啡味重”会影响商品筛选，应优先推荐美式、冷萃、浓缩相关饮品。
- “预算”会影响商品筛选，应优先推荐价格匹配的真实商品。

#### 5.6.2 规格建议原则

AI 推荐结果中的 `options` 用于表达建议规格，而不是直接替用户完成下单。

示例：

```json
{
  "product_id": 12,
  "product_name": "冰美式",
  "reason": "符合冰饮、低糖、无奶和清爽需求",
  "options": {
    "temperature": "ICE",
    "sugar": "LESS",
    "milk": "none"
  }
}
```

前端展示时应让用户理解：

```text
推荐商品：冰美式
建议规格：冰 / 少糖 / 不加奶
推荐理由：符合冰饮、低糖、无奶和清爽需求
```

#### 5.6.3 后续加购原则

Day 8 接入购物车时，不建议 AI 推荐卡片“一键加入购物车”。推荐流程应为：

```text
用户点击推荐卡片的“选择规格”
→ 打开或进入原有商品客制化选择页面
→ 根据 AI options 预填温度、甜度、奶源等选项
→ 用户检查或修改规格
→ 用户确认后加入购物车
```

该方案可以复用现有点单流程，避免 AI 直接替用户下单导致规格错误，也更符合真实小程序点单体验。

## 6. 接口设计

### 6.1 AI 点单接口

请求：

```http
POST /api/ai/order-assistant/
```

请求体：

```json
{
  "user_input": "我想喝一杯冰的、低糖、不要牛奶的咖啡",
  "user_id": 1
}
```

成功响应：

```json
{
  "success": true,
  "data": {
    "intent": "recommend_drink",
    "recommendations": [
      {
        "product_id": 12,
        "product_name": "冰美式",
        "reason": "符合冰饮、低糖、无奶需求",
        "options": {
          "temperature": "ICE",
          "sugar": "LESS",
          "milk": "none"
        }
      }
    ],
    "follow_up_question": "",
    "is_fallback": false
  },
  "message": "推荐成功",
  "error_code": null
}
```

登录失败响应：

```json
{
  "success": false,
  "data": {
    "intent": "login_required",
    "recommendations": [],
    "follow_up_question": "请先登录后再使用 AI 点单助手。"
  },
  "message": "请先登录后再使用 AI 点单助手。",
  "error_code": "LOGIN_REQUIRED"
}
```

### 6.2 商品列表接口

请求：

```http
GET /api/products/
```

用途：

- 获取真实商品列表。
- 用 `id` 匹配 AI 推荐中的 `product_id`。
- 补齐推荐卡片展示所需的价格和图片。

## 8. 页面状态与异常处理

### 8.1 初始状态

页面展示：

- AI 点单助手说明。
- 输入框。
- 发送按钮。
- 示例输入提示。
- 不展示推荐卡片。

### 8.2 加载状态

请求中展示：

```text
正在分析你的口味...
```

按钮置灰或显示 loading。

### 8.3 成功状态

当 `success=true` 且 `recommendations.length > 0`：

- 展示推荐卡片列表。
- 展示后端返回的 `message`。
- 如果 `is_fallback=true`，展示兜底说明。

### 8.4 输入无关内容

当后端返回 `OUT_OF_SCOPE`：

- 不展示商品卡片。
- 展示引导文案：

```text
我主要负责咖啡和饮品点单，可以告诉我你的口味、温度、甜度或预算。
```

### 8.5 未登录

点击发送时如果本地未登录：

- 前端先弹窗，不调用接口。
- 用户点击确认后跳转 `pages/login/login`。

如果接口仍返回 `LOGIN_REQUIRED`：

- 展示同样登录弹窗。
- 作为二次兜底。

### 8.6 网络异常

接口请求失败时：

- 关闭 loading。
- 使用 toast 或提示块展示：

```text
AI 点单暂时不可用，请稍后再试。
```

## 9. UI 展示要求

### 9.1 视觉风格

- 延续现有小程序简洁黑白风格。
- 页面背景以白色或浅灰为主。
- 推荐卡片需要有明确层次，避免信息拥挤。
- 推荐理由需易读，不宜过长。

### 9.2 推荐卡片信息层级

推荐卡片按以下顺序展示：

1. 商品图片
2. 商品名
3. 价格
4. 推荐理由
5. 规格建议
6. 预留操作按钮：选择规格

### 9.3 规格建议展示

将 AI 返回的 options 转换成用户可读文本：

- `temperature: ICE` → 冰
- `temperature: HOT` → 热
- `sugar: LESS` → 少糖
- `sugar: NONE` → 无糖
- `milk: none` → 不加奶
- `milk: oat_milk` → 燕麦奶

如果某字段为空，则不展示该字段。

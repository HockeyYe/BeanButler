# BeanButler 当前版本 PRD

## 1. 文档信息

- 项目名称：BeanButler 咖啡点单系统
- 当前版本范围：现有微信原生小程序 + Django 后端已实现功能
- 文档目的：梳理当前系统能力、业务背景、用户角色、页面流程、接口与数据基础，为后续 3 周 AI 升级开发提供产品基线。
- 当前结论：基础点单链路已经具备，包括商品浏览、规格选择、购物车、结算、下单、订单查询、会员、优惠券、反馈、后台订单处理、库存管理、数据看板和基于历史订单的推荐能力。自然语言 AI 点单助手、咖啡知识问答助手和萃取参数建议尚未在当前版本中实现，属于下一阶段规划。

## 2. 项目背景

BeanButler 面向咖啡店、校园咖啡社团或小型咖啡运营场景，提供一个从用户点单到后台处理的完整系统。项目当前已经包含微信小程序前端和 Django 后端，能够支持用户在线浏览菜单、加入购物车、选择规格、使用优惠券、提交订单，并在后台完成商品、订单、会员、库存和反馈管理。

后续 3 周的升级目标是在这个可运行的点单系统基础上加入 AI 能力，让系统从“普通点单工具”升级为“AI 咖啡点单与知识助手”。因此，当前 PRD 重点记录已有功能边界，避免后续开发时破坏原有业务闭环。

## 3. 项目定位

一句话定位：

面向校园咖啡社团和小型咖啡运营场景的微信小程序点单与后台管理系统，并计划升级为支持 AI 智能点单、咖啡知识问答和萃取参数建议的综合助手。

当前版本定位：

一个已经具备商品、购物车、订单、会员、优惠券、后台管理和数据分析能力的咖啡点单系统。

## 4. 目标用户

### 4.1 普通用户

普通用户通过微信小程序完成登录、浏览商品、选择规格、加入购物车、结算下单、查看订单、使用优惠券和提交反馈。

典型需求：

- 快速找到想喝的咖啡或饮品
- 选择温度、冰量、甜度、咖啡豆等规格
- 查看订单状态和取餐码
- 使用积分和优惠券降低消费成本
- 向商家反馈问题或建议

### 4.2 咖啡社团成员 / 门店运营者

运营者通过 Django 后台维护商品、库存、订单、会员、优惠券、反馈和排班，并查看经营数据。

典型需求：

- 维护商品上下架、价格、图片和可选规格
- 查看并推进订单状态
- 管理会员积分、等级和优惠券
- 查看销售、商品、人群和预测数据
- 处理用户反馈

### 4.3 管理员

管理员负责系统配置、数据维护、后台权限和后续功能迭代。

典型需求：

- 保证系统本地或线上稳定运行
- 管理数据库和后台模型
- 运行推荐任务
- 为后续 AI 接口准备商品、用户和订单数据基础

## 5. 当前已实现功能

### 5.1 微信小程序前端

已实现页面：

- 首页：项目入口页面
- 菜单页：商品分类浏览、推荐区展示、商品规格选择、加入购物车
- 结算页：购物车明细、会员折扣、优惠券选择、取餐时间、提交订单
- 订单页：进行中订单和历史订单查看
- 我的页面：会员信息、等级、积分、优惠券入口、反馈入口、登出
- 登录页：微信登录并写入本地用户态
- 编辑资料页：补充姓名、学号和手机号
- 优惠券页：查看可用优惠券
- 选择优惠券页：结算时选择可用优惠券
- 积分兑换页：使用积分兑换满减券
- 留言反馈页：提交反馈并查看后台回复

前端公共能力：

- `frontend/utils/api.js` 封装本地 Django 和腾讯云托管两种请求方式
- 支持 `local` / `cloud` 环境切换
- 购物车、优惠券、用户信息使用 `app.globalData` 和本地缓存维护
- 自定义底部 TabBar 支持首页、点单、订单、我的四个主入口

### 5.2 商品与菜单

后端商品模型支持：

- 商品名称
- 分类：意式咖啡、手冲滴滤、无咖啡因、招牌特调
- 价格
- 描述
- 商品图片
- 上下架状态
- 是否在推荐区展示
- 是否允许选择咖啡豆
- 可关联咖啡豆库存
- 可选温度 / 冰量
- 可选甜度

小程序菜单页支持：

- 从 `/api/products/` 拉取可售商品
- 按商品分类展示
- 根据 `is_recommended` 展示“为你推荐”区域
- 自动修复商品图片 URL
- 打开商品规格弹窗
- 选择咖啡豆、冰量、甜度、数量
- 加入购物车

### 5.3 购物车与结算

当前购物车能力：

- 商品加入购物车
- 同一商品不同规格可作为不同条目
- 购物车数量增减
- 清空购物车
- 展示购物车件数和金额
- 结算前校验登录状态
- 结算前校验用户资料完整性

当前结算能力：

- 展示购物车明细
- 计算商品原价小计
- 根据会员等级计算折扣
- 选择并校验优惠券
- 计算最终应付金额
- 生成预计获得积分
- 选择取餐时间，包含“立即取餐”和按 15 分钟生成的时间段
- 提交订单到后端 `/api/orders/`

会员折扣规则：

- Bronze / 黄金会员 V1：无折扣
- Silver / 白金会员 V2：9 折
- Gold / 黑金会员 V3：85 折

注意：

结算页当前会尝试调用 `/api/orders/clear-ongoing/` 清理进行中订单，但后端路由中暂未看到该接口。当前代码会在清理失败时继续下单，因此主下单流程仍可执行，但该接口属于后续需要补齐或移除的技术债。

### 5.4 订单

后端订单能力：

- 创建订单：`POST /api/orders/`
- 查询会员订单：`GET /api/orders/list/?member_id=xxx`
- 自动生成订单号，格式为 `YYYYMMDD + 3 位序号`
- 自动生成取餐码
- 支持会员订单，也允许匿名订单结构
- 保存订单总价、取餐时间、取餐码、状态和商品 JSON 快照
- 同时写入结构化 `OrderItem`，方便后续库存、统计和推荐使用
- 优惠券在下单时核销

订单状态：

- `PENDING`：待确认
- `CONFIRMED`：已确认
- `IN_PROGRESS`：制作中
- `READY`：待取餐
- `PICKED_UP`：已取餐

小程序订单页支持：

- 登录后拉取后端订单
- 区分进行中订单和历史订单
- 展示订单号、状态、创建时间、取餐码、金额和商品明细
- 订单详情弹窗展示关键订单信息

后台订单能力：

- Django Admin 中可查看订单列表
- 支持批量确认、开始制作、备好、完成取餐
- 取餐完成时发放积分并更新会员等级
- 自定义订单工作台可查看未完成订单并更新状态

### 5.5 会员

后端会员模型支持：

- 真实姓名
- 昵称
- 学号
- 头像 URL
- 微信 OpenID
- 手机号
- 会员等级
- 积分
- 加入时间

会员接口：

- 微信登录：`POST /api/login/`
- 查询会员：`GET /api/member/?member_id=xxx`
- 更新资料：`POST /api/member/update-profile/`
- 上传头像：`POST /api/upload_avatar/`

会员规则：

- 新用户通过微信 code 换取 OpenID 并创建会员
- 用户资料不完整时提示补充姓名、学号和手机号
- 积分达到门槛后可升级会员等级
- 消耗积分兑换优惠券不会导致会员等级下降

等级门槛：

- Bronze：默认等级
- Silver：100 积分
- Gold：300 积分

### 5.6 优惠券与积分兑换

优惠券模型支持：

- 所属会员
- 券名称
- 最低消费门槛
- 折扣金额
- 到期日
- 是否已使用
- 发放时间

优惠券接口：

- 查看可用优惠券：`GET /api/coupons/?member_id=xxx`
- 查看兑换选项：`GET /api/coupons/options/`
- 积分兑换优惠券：`POST /api/coupons/redeem/`

当前固定兑换选项：

- 50 积分兑换 MOP 5 满减券，满 MOP 30 可用
- 100 积分兑换 MOP 10 满减券，满 MOP 60 可用
- 180 积分兑换 MOP 20 满减券，满 MOP 100 可用

结算页会根据当前购物车金额判断优惠券是否达到使用门槛，并在下单成功后由后端核销优惠券。

### 5.7 用户反馈

反馈接口：

后台数据看板接口：

当前看板能力：

- 与昨日对比变化
- 指定时间范围内营业额、订单数、客单价

- 会员升级预估

### 5.10 当前推荐功能

当前系统已有“推荐”能力，但它不是自然语言 AI 点单助手，而是基于历史订单的协同过滤推荐。

后端推荐能力：

- 推荐接口：`GET /api/recommendations/?member_id=xxx`
- 如果会员已有推荐结果，则返回个性化推荐
- 如果没有会员或没有推荐结果，则回退到全局热门商品
- 推荐结果限制为可售商品

推荐任务：

- 管理命令：`python manage.py run_cf_recommender`
- 基于已完成订单构建用户购买记录
- 通过商品-会员矩阵计算商品相似度
- 结合商品热度生成 Top 3 推荐
- 推荐结果写入 `Recommendation`
- 运行记录写入 `RecommenderRunLog`
- 后台可手动触发推荐任务

小程序菜单页会调用推荐接口，并在菜单页展示推荐商品。

## 6. 当前核心用户流程

### 6.1 普通点单流程

1. 用户进入小程序。
2. 用户登录或浏览菜单。
3. 用户进入菜单页。
4. 前端请求 `/api/products/` 获取商品列表。
5. 用户选择商品并打开规格弹窗。
6. 用户选择咖啡豆、冰量、甜度和数量。
7. 用户加入购物车。
8. 用户进入结算页。
9. 系统校验登录状态和资料完整性。
10. 系统展示会员折扣、可用优惠券和取餐时间。
11. 用户提交订单。
12. 后端创建订单和订单项。
13. 用户进入订单页查看订单状态和取餐码。
14. 后台处理订单并推进状态。
15. 用户取餐后，后台标记为已取餐并发放积分。

### 6.2 优惠券使用流程

1. 用户在“我的”页面进入积分兑换。
2. 前端请求 `/api/coupons/options/` 获取可兑换券。
3. 用户使用积分兑换满减券。
4. 后端扣减积分并创建优惠券。
5. 用户结算时进入选择优惠券页。
6. 前端请求 `/api/coupons/` 获取可用优惠券。
7. 用户选择满足门槛的优惠券。
8. 结算页计算优惠券抵扣金额。
9. 提交订单后，后端核销该优惠券。

### 6.3 反馈流程

1. 用户进入“我的”页面。
2. 用户进入留言反馈页。
3. 用户提交反馈内容。
4. 后端创建反馈记录。
5. 管理员在后台回复。
6. 用户再次进入留言页查看回复内容。

### 6.4 推荐商品流程

1. 管理员运行推荐任务或后台触发推荐。
2. 系统基于已完成订单生成会员推荐结果。
3. 用户打开菜单页。
4. 前端请求 `/api/recommendations/`。
5. 如果存在会员推荐结果，展示个性化推荐。
6. 如果不存在，展示热门商品推荐。
7. 用户可从推荐商品进入规格选择并加入购物车。

## 9. 非功能要求与当前实现

### 9.1 环境配置

当前前端支持本地和云端两种请求方式：

- 本地：`http://127.0.0.1:8000`
- 云端：腾讯云托管地址

后端通过 `.env.example` 提供环境变量示例，包括：

- `DJANGO_DEBUG`
- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `DJANGO_SECRET_KEY`
- `WECHAT_APP_ID`
- `WECHAT_APP_SECRET`

### 9.2 数据安全

当前 README 已提示真实 `.env` 不应提交到 GitHub。

需要关注：

- `members/views.py` 中存在微信 AppID 和 AppSecret 的默认值，后续应改为仅从环境变量读取，避免密钥暴露。
- AI API Key 后续必须只放在后端环境变量中，不能出现在小程序前端。

### 9.3 稳定性

当前后端接口大多返回统一的 `success` 字段，但错误结构尚未完全统一。

后续 AI 接口应统一返回：

```json
{
  "success": true,
  "data": {},
  "message": "",
  "error_code": null
}
```

## 12. 后续 AI 升级方向

### 12.1 AI 智能点单助手

目标：

用户输入自然语言需求，例如“我想喝冰的、低糖、不加奶的咖啡”，系统根据真实商品菜单返回推荐商品。

依赖当前系统：

- 商品接口 `/api/products/`
- 商品 ID、名称、分类、价格、规格字段
- 购物车加入逻辑

第一阶段接口建议：

- `POST /api/ai/order-assistant/`

核心要求：

- AI API Key 只放后端环境变量
- AI 返回结构化 JSON
- 推荐必须匹配真实 `product_id`
- 异常时返回前端可处理的 fallback

### 12.2 咖啡知识问答助手

目标：

用户可以询问咖啡知识，例如“手冲偏酸怎么办”“意式萃取太苦怎么调”。

依赖当前系统：

- 新增本地咖啡知识库
- 后端新增检索和问答接口
- 小程序新增问答页面

第一阶段接口建议：

- `POST /api/ai/coffee-qa/`

### 12.3 萃取参数建议

目标：

用户输入粉量、水量、水温、时间和口感问题，系统输出可能原因和调整建议。

可作为咖啡知识问答助手的一个场景，不必第一版单独做复杂页面。
